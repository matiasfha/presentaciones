6 Que es React?

  • Publicado el 2003, es una librería para crear interfaces de forma declarativa
  • Implementó nuevas ideas sobre como crear aplicaciones web.
  • Propone usar la composición de pequeñas y aisladas unidades para crear interfaces complejas.
  • Propone utilizar un nuevo concepto: VDOM
  • Define la interfaz como una función del estado
  • Permite utilizar una forma declarativa para definir la interfaz.
  • Propone una nueva sintaxis sobre Javascript.
  • Permite su adopción de forma gradual.

Es una librería javascript para crear interfaces de forma declarativa. Te permite componenr interfaces complejas desde pequeñas y aisladas piezas de codigo llamadas “componentes”.

Por medio de una sintax “similar” al html React te permite declarar que quieres mostrar en pantalla, sin preocuparte de como esta interfaz se despliega.

React define que la interfaz es una función del estado, es decir, lo que se despliega en pantalla es el resultado de aplicar cierta lógica, cambios o cálculos sobre un conjunto de datos.

Esta función del estado es una función pura, es decir, es predecible ya que siempre que reciba los mismos datos, la interfaz creada será la misma.

React, se encarga de la creación y manipulación de la interfaz, así como también del como los datos se “conectan” con la interfaz.

Es opensource pero mantenido y dirigido por facebook.

Lanzado el 2013 fue en su momento controversial por la forma en que se proponía crear aplicaciones, cambiando el concepto de “responsabilidad única”, desde las ideas de separar logica, presentación y representación a cambiar por resposabilidad.

8 ¿Qué es VDOM?

  • Un concepto que define la creación de una representación ideal o virtual de la interfaz
  • Esta representación es la que se manipula y se sincroniza con el DOM reconciliacion
  • Permite indicarle a React en que estado se quiere la UI sin indicar como modificar el DOM.
  • Permite el renderizado incremental del DOM (React Fiber)

9 Como funciona VDOM

  • Al renderizar un componente, todos los objetos del VDOM son actualizados.
  • React compara el nuevo arbol VDOM con el estado anterior y obtiene la diferencia
  • Utiliza esta diferencia para actualizar el DOM solo en esos elementos
  • Every DOM object have a VDOM representation
  • Have the same properties as the DOM but without the power of changing the UI.
  • Manipular el DOM es lento, el VDOM es mas rápido.

20 ¿Cómo saber que es componente?

  • Decision similar a crear una nueva función.
  • Single Responsibility Principle
  • Reusabilidad
  • ⚠️ No optimices prematuramente
  • ⚠️no crees abstracciones innecesarias

Es una decisión similar a decidir cuando crear o no una nueva funcion. una de estas ténicas y probablemente la más usada es el principio de responsabilidad unica, es decir, el componente deberia hacer una y solo una cosa. Si crece más allá de ese responsabilidad, entonces deberias descomponerlo. Otro punto es la reusabilidad. ¿Usarás este componente en más de un lugar? si la respuesta es si, entonces hay que descoponenerlo y crear uno en particular

Cuidado No ha que hacer optimizaciones prematuras y crear abstracciones innecesarias antes de tiempo, si bien ambas ténicas son completamente validas y ciertas, lo es también que si: no estás seguro de si se reutilizará, no tiene mucho sentido dividir un componente a menos claro, que sea por cuestiones de mantenibilidad.

21 Relacion data y UI

  • UI es la forma de comunicarse con el usuario
  • UI es una representacion de los datos
  • Divide tu interfaz en componentes donde cada componente represente una parte de los datos

La interfaz es la forma que tiene la aplicación web de mostrar información al usuario, de presentar el Modelo de una forma que el usuario pueda entender, pero si el modelo de datos fue diseñado adecuadamente podrás ver que muchas veces la interfaz es muy similar y se “mapea” correctamete con el Modelo de datos. Esto es por que los modelos de datos y la interfaz se adhieren al mismo tipo de arquitectura de la información. Divide tu interfaz en componentes, donde cada componente representa una parte de los datos.

24 Sobre JSX

  • JS permite manipular el DOM, por medio de APIs
  • React es declarativo. No manipulamos el DOM directamente
  • React ofrece una API para la creación de componentes
  • React soporta múltiples plataformas una de ellas es el DOM mediante ReactDOM.
 const rootElement = document.createElement('div')
 rootElement.setAttribute('id','root')
 document.body.append(rootElement)

 const element = document.createElement('div')
 element.textContent = 'Hola Mundo'
 element.className = 'container'
 rootElement.append(element)

javascript permite manipular el DOM y agregar, quitar o actualizar el arbol de componentes, javascript ofrece varias api que permiten acceder a los elementos html para ser modificados. Esta es la forma imperativa de trabajar con interfaces y es lo que jQuery (y otros) proponía.

React es declarativo, por lo que no tenemos que manipular el DOM directamente, si no, sólo definifnr o declarar que queremos que se muestre en el DOM.

Para eso React ofrece una serie de API que permiten crear los componentes que desplegaran información. En el fondo React utiliza las mismas api imperativas para manipular el DOM que han existido dsde que JS se inventó, pero esto es para que tu no tengas que hacerlo directamente.

One important thing to know about React is that it supports multiple platforms (for example, native and web). Each of these platforms has its own code necessary for interacting with that platform, and then there’s shared code between the platforms.

With that in mind, you need two JavaScript files to write React applications for the web:

React: responsible for creating React elements (kinda like document.createElement()) ReactDOM: responsible for rendering React elements to the DOM (kinda like rootElement.append())

28 Arrays

¿Qué ocurre si se hace agrega un nuevo elemento a la lista?

If you re-render that list with an added item, React doesn’t really know whether you added an item in the middle, beginning, or end. And the same goes for when you remove an item (it doesn’t know whether that happened in the middle, beginning, or end either).

In this example, it’s not a big deal, because React’s best-guess is right and it works out ok. However, if any of those React elements represent a component that is maintaining state, that can be pretty problematic

29 Ejemplo

  • Solution

You’ll feel this problem with anything that maintains state over time. The input is a good example. Any component that’s maintaining state will also experience this problem, so the key prop is really important. Whenever you see that warning, you want to make sure that you provide a unique key for each element of the array that you’re rendering.

4:04 Just randomly generating a key, right here, is not going to cut it because it needs to be consistent for the React element that’s representing that particular item. Having an id is preferable because that’s something that can be unique for each one of these items.

4:20 If you don’t have an id, then maybe you could generate an id for the items before you go to render them, or you could use something else about the items that are unique but don’t try to use the index of that item because that’s the default behavior.

4:33 If we do item and then index and we pass the index, then we’ll get the exact same behavior that we had before, which is not the behavior that we want. Don’t use the index. That’ll silence the warning but not get rid of the problem. You need to do some unique identifier for the particular item that you’re trying to represent when you’re rendering an array.
